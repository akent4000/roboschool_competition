# Архитектурное решение: автономная навигация квадропеда Aliengo

## Общее описание

Решение реализует программные модули для автономной навигации четвероногого робота Aliengo в неизвестной среде. Задача: исследовать комнату, обнаружить 5 целевых объектов (backpack, bottle, chair, cup, laptop) и посетить их в заданной организаторами последовательности.

**Предоставлено организаторами:** симуляционная среда (Isaac Gym), обученная RL-политика локомоции (трансляция скоростных команд в движение ног), интерфейс робота (`AliengoRobotInterface` с методами `set_speed`, `get_state`, `get_camera`), RGB-D камера, ROS 2 bridge, шаблон контроллера с заглушкой `get_found_object_id → None`, система логирования.

**Реализовано командой:** детекция объектов, SLAM-навигация, исследование пространства, планирование пути, управление движением, подтверждение посещения, визуализация — всё, что находится внутри блоков `USER PARAMETERS` / `USER CONTROL LOGIC` файла `main_controller.py`, а также отдельные модули `slam.py` и `visualizer.py`.

## Архитектура решения команды

### Общая схема

```
┌─────────────────────────────────────────────────────────────┐
│               ЛОГИКА УПРАВЛЕНИЯ (команда)                   │
│           (оркестрация, конечный автомат состояний)          │
├───────────┬────────────────┬──────────────┬─────────────────┤
│  YOLO     │  SLAM-модуль   │  Менеджер    │  Dashboard      │
│  детектор │  (карта +      │  очереди     │  визуализатор   │
│           │   навигация)   │  объектов    │                 │
├───────────┴────────────────┴──────────────┴─────────────────┤
│          API организаторов: set_speed(vx, vy, wz)           │
│              get_state() / get_camera()                      │
│      (Isaac Gym + RL-политика + сенсоры — чёрный ящик)      │
└─────────────────────────────────────────────────────────────┘
```

Команда работает поверх API организаторов: получает сенсорные данные (RGB, depth, IMU, скорости тела) и отправляет обратно скоростные команды `(vx, vy, wz)`. Всё, что ниже этого интерфейса (локомоция, физика, рендеринг) — зона ответственности организаторов.

### Поток данных

```
API организаторов: get_state() / get_camera()
        │
        ▼
  ┌─── Perception (команда) ───┐
  │  YOLO-детекция              │──► 2D bbox + класс + confidence
  │  Depth sampling             │──► 3D позиция объекта в мире
  │  Bbox fallback              │──► оценка дистанции при depth > 4м
  └────────────────────────────┘
        │
        ▼
  ┌─── Memory (команда) ───────┐
  │  known_objects              │  карта известных объектов (class → world pos)
  │  sight_counts               │  аккумулятор наблюдений (фильтрация шума)
  │  visited_positions          │  исключение уже посещённых зон
  └────────────────────────────┘
        │
        ▼
  ┌─── SLAM (команда) ─────────┐
  │  Odometry                   │  интеграция скоростей → глобальная поза
  │  OccupancyGrid              │  log-odds карта 40×40м, разрешение 5см
  │  FrontierExplorer           │  поиск границ исследованного пространства
  │  PathPlanner (A*)           │  планирование пути с учётом препятствий
  │  PurePursuit                │  генерация (vx, vy, wz) команд
  └────────────────────────────┘
        │
        ▼
  API организаторов: set_speed(vx, vy, wz) → RL-политика → движение
```

## Реализованные модули

### 1. Пайплайн подготовки данных и обучения детектора

Полный цикл от сбора данных в симуляторе до готовой YOLO-модели реализован тремя скриптами.

#### Сбор датасета (`scripts/collect_dataset.py`)

Автоматический сбор размеченных RGB-кадров из симулятора:
- Робот телепортируется на сетку точек обзора вокруг каждого из 5 объектов
- **Сетка viewpoints:** 7 дистанций (1.0–4.0м) × 12 азимутов (каждые 30°) × 3 поворота камеры (центр, ±25°) = до 252 кадров на объект
- После телепортации физика стабилизируется (40 шагов ≈ 0.8с), затем делается снимок с бортовой камеры
- **Негативные примеры:** кадры с камерой, направленной от объекта, + случайные позиции вдали от всех объектов (до 60 кадров)
- **Разные раскладки комнат:** поддержка нескольких seed — каждый seed порождает уникальное расположение объектов и препятствий (Isaac Gym ограничение: один PhysX на процесс, поэтому seed запускаются в отдельных subprocess)
- **Два формата выхода:**
  - Per-class folders (для классификации)
  - YOLO detection format (`images/train/` + `labels/train/` + `data.yaml`) — разметка bbox через segmentation buffer симулятора (pixel-perfect, с учётом окклюзии)

#### Обучение модели (`scripts/train_yolo.py`)

Fine-tuning YOLOv8n на собранном датасете:
- Автоматическое разбиение train/val (по умолчанию 80/20) с фиксированным seed
- **Аугментации, настроенные под sim-to-sim:**
  - HSV: h±1.5%, s×0.5, v×0.3
  - Геометрия: ±10° поворот, ±10% сдвиг, ±30% масштаб
  - Mosaic + MixUp (0.1)
  - Горизонтальный flip: 50%, вертикальный flip отключён (объекты не бывают перевёрнутыми)
- Early stopping с patience=20 эпох
- Автоматическая валидация и экспорт в ONNX после обучения
- Поддержка возобновления прерванного обучения (`--resume`)

#### Просмотр датасета (`scripts/view_dataset.py`)

Интерактивный визуализатор для контроля качества собранных данных:
- Отображение RGB-кадров с наложением YOLO bounding boxes (полупрозрачные + контур + подпись класса)
- Фильтрация по классу (`--class backpack`) и наличию разметки (`--only_labeled`)
- Клавиатурное управление: навигация (←/→), переключение bbox (B), имя файла (F)
- Статистика: номер кадра, количество объектов, распределение по классам

### 2. Детекция объектов в runtime (YOLO + 3D-локализация)

**Модель:** YOLOv8n, обученная пайплайном выше на синтетических данных из симулятора.

**Принцип работы:**
- Инференс запускается каждые 5 шагов симуляции (~10 Гц) для баланса между точностью и производительностью
- Базовый порог confidence: 60%, для chair повышен до 82% из-за высокого уровня false-positive
- Стабильная детекция: объект должен быть обнаружен 10 раз подряд перед логированием в судейскую систему

**3D-локализация (двойная стратегия):**
- **Первичная (depth-камера):** выборка глубины в патче 30×30 пикселей вокруг центра bbox с медианной фильтрацией, перепроецирование RGB-координат в depth-кадр через intrinsics обеих камер, преобразование pixel + depth → world coordinates с учётом текущей позы робота
- **Fallback (pinhole-модель):** при дальности > 4м (за пределами depth-сенсора) расстояние оценивается по высоте bounding box через формулу `dist = (real_height × fx) / bbox_height`

### 3. Память объектов

Трёхуровневая система запоминания позиций обнаруженных объектов:
- **Depth-confirmed:** объект сразу сохраняется в `known_objects` при наличии достоверной глубины — высокая точность позиции
- **Sight accumulator:** при bbox-оценке (без глубины) требуется минимум 3 наблюдения с позициями в радиусе 2м — позиция усредняется по мере накопления
- **Stale eviction:** при depth-подтверждении нового объекта старые записи в радиусе 1.5м удаляются — борьба с ошибками классификации, когда YOLO путал классы

### 4. SLAM-модуль (`slam.py`)

Полностью написан командой с нуля, без использования ROS Navigation Stack или других готовых SLAM-фреймворков.

#### Одометрия (OdometryTracker)
- Dead-reckoning: интеграция измеренных скоростей тела `(vx, vy, wz)` из API организаторов в глобальную позу `(x, y, θ)`
- Нормализация угла через `atan2(sin, cos)` для предотвращения накопления ошибки

#### Карта занятости (OccupancyGrid)
- Размер: 40×40м, разрешение: 5 см/ячейка (800×800 ячеек)
- Обновление: log-odds модель (`L_OCC=0.85`, `L_FREE=-0.4`, пороги ±1.5)
- Векторизованная проекция глубины: обработка 848 столбцов depth-изображения в горизонтальных полосах (каждый 3-й столбец)
- DDA raycasting для пометки свободного пространства вдоль лучей
- Inflated obstacle mask через OpenCV dilation для безопасного планирования
- Частота обновления: каждые 10 шагов (~5 Гц)

#### Исследование фронтов (FrontierExplorer)
- Фронт = свободная ячейка, граничащая с неизвестным пространством
- Кластеризация фронтов через flood-fill для борьбы с шумом
- Стратегия выбора `least_explored`: оценка = доля неизвестного в окрестности + бонус за размер − штраф за расстояние
- Фильтрация: фронты в радиусе 1.5м от посещённых объектов исключаются
- Частота пересчёта: каждые 50 шагов (~1 Гц)

#### Планировщик пути (A* с штрафом за повороты)
- 8-направленный A* на инфлированной карте
- Штраф за повороты: `LAMBDA_TURN=0.3 × angle_change` — поощряет плавные пути
- Line-of-sight smoothing: удаление избыточных waypoints
- Densification: вставка промежуточных точек с шагом 0.2м для плавного управления
- Ограничение: максимум 200 000 итераций

#### Навигационный контроллер (Pure Pursuit + реактивное избежание)
- **Pure Pursuit:** lookahead = 0.8м, адаптивная скорость по кривизне
- **Straight-line boost:** до 1.20 м/с на прямых участках (ошибка < 8°), нормальная скорость 0.65 м/с
- **Obstacle avoidance:** стоп при 0.55м, замедление 0.55–1.0м, латеральный уход при боковых препятствиях < 0.6м
- **Anti-stuck:** sticky backing (1.2с), hard turn (0.6с), random walk (при отсутствии фронтов 2с)
- **Low-pass фильтр:** сглаживание угловой скорости (α=0.3), мёртвая зона ±4°

### 5. Конечный автомат состояний

Контроллер верхнего уровня управляет режимами движения робота:

| Режим | Условие | Поведение |
|-------|---------|-----------|
| `warmup_stop` | `t < 3.5с` | Робот стабилизируется после спавна |
| `navigate` | Активная навигация | SLAM-команды `(vx, vy, wz)` |
| `confirm_wait` | В радиусе 0.8м от цели | Полная остановка на 2.1с |
| `backup` | После подтверждения | Движение назад 1.5с |
| `all_done_stop` | Все объекты посещены | Полная остановка |

### 6. Pass-by подход (оптимизация маршрута)

Когда известна позиция следующего объекта в очереди, робот подходит к текущей цели не в лоб, а со смещением 0.4м в сторону следующей цели. Выбирается сторона с минимальным суммарным путём (робот → точка подхода → следующая цель). Это сокращает суммарное время, особенно при последовательно расположенных объектах.

### 7. Подтверждение посещения (двойной механизм)

- **YOLO-подтверждение:** объект виден камерой + робот в радиусе 0.8м + стоит неподвижно 2.1с. Если YOLO теряет объект в свежем кадре — подтверждение отменяется
- **Proximity-подтверждение:** альтернатива при подходе сбоку, когда камера не видит объект — робот в радиусе 0.8м от запомненной мировой позиции + стоит 2.1с

### 8. Визуализация (`visualizer.py`)

Реальновременная отладочная панель через OpenCV (заменяет базовый `_CameraRenderer` организаторов):
- **Левая часть:** RGB с наложением YOLO bounding boxes, подсветка текущей цели, шкала глубины
- **Правая часть:** карта занятости с роботом, спланированным путём, waypoints, фронтами исследования, навигационной целью, известными объектами, зонами посещения
- **Строка статуса:** скорости, прогресс очереди, таймер подтверждения, одометрия

## Сильные стороны решения

### 1. Робастная 3D-локализация с двойным fallback
Система не теряет объекты за пределами дальности depth-сенсора (4м). При отсутствии глубины расстояние оценивается по размеру bounding box, а позиция уточняется при повторных наблюдениях. Это критично для эффективного исследования — робот может начать движение к далёкому объекту ещё до входа в зону depth-камеры.

### 2. Полноценный 2D SLAM без внешних зависимостей
Вся SLAM-подсистема (одометрия, log-odds карта, frontier exploration, A* планировщик, Pure Pursuit контроллер) реализована с нуля без ROS Navigation Stack или других тяжёлых фреймворков. Это обеспечивает полный контроль над поведением, лёгкую отладку и отсутствие проблем совместимости с предоставленным окружением.

### 3. Интеллектуальная память объектов с self-correction
Система не просто запоминает позиции объектов — она накапливает наблюдения, усредняет позиции, автоматически вытесняет ошибочные классификации при поступлении более достоверных данных. Это минимизирует влияние ложных срабатываний YOLO и позволяет роботу уверенно направляться к ранее замеченным объектам.

### 4. Адаптивная стратегия навигации
Контроллер плавно переключается между тремя режимами: frontier-based exploration (когда цель неизвестна), направленное движение к запомненному объекту и подтверждение на месте. Pass-by оптимизация дополнительно сокращает суммарное время маршрута при последовательном посещении объектов.

### 5. Многоуровневая защита от застревания
Три уровня anti-stuck: sticky backing (предотвращает осцилляцию перед препятствием), hard turn (разворот при длительном застревании), random walk (при отсутствии фронтов). Это критически важно для робастной работы, где застревание = потеря времени и очков.

### 6. Плавное управление движением
Pure Pursuit с lookahead, кривизно-адаптивная скорость, low-pass фильтрация угловой скорости и мёртвая зона — всё вместе обеспечивает плавную траекторию без рывков, что важно для стабильности четвероногого робота. Straight-line boost до 1.2 м/с на прямых участках ускоряет перемещение между далёкими точками.

### 7. Двойное подтверждение посещения
Комбинация YOLO + proximity подтверждения исключает ситуации, когда робот не может подтвердить объект из-за неудачного угла камеры. Стабильная детекция (10 последовательных кадров) отсеивает случайные срабатывания перед логированием в судейскую систему.

### 8. Эффективное использование вычислительных ресурсов
Каждый модуль работает со своей оптимальной частотой:
- YOLO-детекция: ~10 Гц (каждые 5 шагов)
- Обновление карты: ~5 Гц (каждые 10 шагов)
- Фронты и пути: ~1 Гц (каждые 50 шагов)
- Dashboard: ~17 Гц (каждый 3-й шаг)

Такое частотное разделение обеспечивает работу в реальном времени без перегрузки GPU.

### 9. Полный пайплайн данных для детектора
Весь цикл от сбора данных до обученной модели автоматизирован: сбор через телепортацию в симуляторе с pixel-perfect разметкой из segmentation buffer, автоматическое разбиение train/val, обучение с аугментациями под sim-to-sim, экспорт в ONNX. Интерактивный просмотрщик позволяет быстро оценить качество данных. Разные seed обеспечивают разнообразие раскладок комнат.

### 10. Полная визуализация для отладки
Dashboard в реальном времени показывает всё состояние системы: что видит камера, результаты YOLO, состояние карты, спланированный путь, доступные фронты. Это кардинально ускоряло разработку и позволяло быстро находить проблемы в навигации.

## Технологический стек (реализация команды)

| Компонент | Технология |
|-----------|-----------|
| Сбор датасета | Телепортация + segmentation buffer (Isaac Gym) |
| Обучение детектора | YOLOv8n (ultralytics), ONNX export |
| Детекция в runtime | YOLOv8 inference + depth reprojection |
| SLAM и навигация | Собственная реализация (numpy, OpenCV) |
| Планирование пути | A* с turn penalty |
| Управление движением | Pure Pursuit + reactive avoidance |
| Визуализация | OpenCV (cv2) |
